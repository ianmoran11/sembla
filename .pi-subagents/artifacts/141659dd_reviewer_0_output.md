## Review

### Blocking issues

- **Blocker: the change redefines the CPU oracle’s reduction semantics instead of preserving the existing oracle contract.** The pre-change/runtime contract is one sequential ascending-row pass (`docs/prds/0005-expr-eval.md:31-36`), and `DESIGN.md:325-330` says the CPU interpreter remains the semantics oracle and the numeric contract is unchanged. The workspace now changes grouped sums to two half-folds plus a merge (`crates/sembla-runtime/src/eval.rs:1544-1546`, `crates/sembla-runtime/src/eval.rs:1632-1645`), input sums likewise (`crates/sembla-runtime/src/eval.rs:961-990`), and wire-output sums likewise (`crates/sembla-runtime/src/executor.rs:862-904`). This is observably different, not merely an implementation refactor: the modified test explicitly records that `[1e16, 1, -1e16, 1]` now yields `0.0` where the prior sequential fold yields `1.0` (`crates/sembla-runtime/tests/eval.rs:678-710`). That contradicts acceptance criterion 1’s no-CUDA behavior being “unchanged” (`docs/prds-npe-path/0008-cuda-backend.md:65-67`) and leaves two authoritative numeric contracts in conflict. The CUDA tree must match the established CPU oracle, or the semantic-contract change must be explicitly approved and the governing documentation/compatibility expectations updated before this PRD can be accepted.

- **Blocker: CUDA does not preserve the CPU evaluator’s edge-case error order for arbitrary validated models.** CPU staging evaluates each transition’s guard, hazard, and claim columns together in transition declaration order (`crates/sembla-runtime/src/executor.rs:650-680`). CUDA launches guard/hazard kernels for *all* transitions first, scans all candidate errors, and only then evaluates claim expressions (`crates/sembla-cuda/src/backend.rs:457-520`; generated stages at `crates/sembla-cuda/src/codegen.rs:993-1025`). Therefore a valid model whose earlier transition has an overflowing integer claim key and whose later transition has an overflowing guard/hazard returns the earlier claim error on CPU but the later candidate error on CUDA. The existing claim-overflow fixture proves such checked claim expressions are in the validated fragment (`crates/sembla-cuda/tests/gpu_semantics.rs:435-451`), but it does not combine competing errors and so does not cover this ordering mismatch. This violates the requirement for the same evaluator operations and edge cases (`docs/prds-npe-path/0008-cuda-backend.md:37-40`) and prevents claiming support for arbitrary validated v0.1 models.
