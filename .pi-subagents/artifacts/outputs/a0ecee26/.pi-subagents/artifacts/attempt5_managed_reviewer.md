## Review

**Verdict: REVISE**

### Blockers

- **Blocker — aggregate staging still changes the CPU oracle's first-error order in all three requested residual cases.** The CPU evaluates transition expressions lazily in syntax order: arithmetic evaluates `lhs` before `rhs` (`crates/sembla-runtime/src/eval.rs:1226-1238`), and each box evaluates guard, hazard, claims, resolution, then winning effects in declaration order (`crates/sembla-runtime/src/executor.rs:633-718`, `720-757`). CUDA instead:
  - collects aggregates found anywhere in a transition expression (`crates/sembla-cuda/src/codegen.rs:219-261`, `293-358`) and launches every first-use aggregate for that rule **before** the transition kernel (`crates/sembla-cuda/src/backend.rs:419-465`);
  - launches every active effect-only aggregate before `sembla_prepare_effects` (`crates/sembla-cuda/src/backend.rs:579-658`);
  - rebuilds every output aggregate before launching any output field (`crates/sembla-cuda/src/backend.rs:677-754`).

  Consequently, a validated single guard with an overflowing scalar left operand and an overflowing aggregate in its right operand reports the scalar error on CPU but the aggregate error on CUDA. The same counterexample exists with an earlier scalar effect and a later effect aggregate, and with an earlier output field/filter scalar error and a later output aggregate. Existing coverage tests only **different transitions** (`crates/sembla-cuda/tests/gpu_semantics.rs:447-462`, `570-589`) and uncomplicated active/inactive effect aggregates (`crates/sembla-cuda/tests/gpu_semantics.rs:369-386`, `593-627`); it does not cover same-transition scalar/aggregate order or competing effect/output failures. This violates the exact evaluator/edge-case requirement at `docs/prds-npe-path/0008-cuda-backend.md:37-40` for valid v0.1 models.

- **Blocker — multi-box execution orders later-box scheduling before earlier-box effects, unlike the CPU.** CPU calls `stage_box` sequentially and propagates its error before moving to the next box (`crates/sembla-runtime/src/executor.rs:298-309`); `stage_box` includes conflict resolution and effect-column evaluation (`crates/sembla-runtime/src/executor.rs:718-757`). CUDA's global transition loop stages every box first (`crates/sembla-cuda/src/backend.rs:419-534`), performs global resolution next (`535-578`), and evaluates effects only afterward (`579-658`). Thus, for a valid two-box model where box 0 has a winning effect overflow and box 1 has a guard overflow, CPU reports box 0's effect error while CUDA records box 1's guard error and skips effect preparation. Multi-box execution is explicitly in scope (`docs/prds-npe-path/0008-cuda-backend.md:17-20`) and criterion 3 includes `sir_policy` (`:70-72`). No cross-box competing-error test is present.

- **Blocker — generated scalar expressions do not preserve CPU expression/error sequencing and can invoke C++14 undefined behavior.** CPU evaluates whole value columns subtree-first (`crates/sembla-runtime/src/eval.rs:1237-1273`): all rows of `lhs` are completed before `rhs`. CUDA emits checked subexpressions inline as function-call arguments and binary operands (`crates/sembla-cuda/src/codegen.rs:614-697`) and evaluates a complete expression independently per row in the transition kernel (`:994-1022`). Therefore, if `lhs` first overflows at row 1 while `rhs` overflows at row 0, CPU fails from `lhs` row 1 without evaluating `rhs`, while CUDA marks row 0 and the candidate scan selects it. In addition, a valid expression with overflowing checked helpers in both sibling operands causes both calls to modify `*error` without sequencing under the explicitly requested C++14 mode (`crates/sembla-cuda/src/backend.rs:133-140`), which is undefined behavior. This is both an exact-error semantic defect and a generated-source safety defect.

- **Blocker — later failure phases are interleaved/parallel where the CPU is declaration-ordered.** CPU evaluates all live effect columns before global double-write detection (`crates/sembla-runtime/src/executor.rs:311-315`, `720-757`), but CUDA detects ownership collisions while processing each effect, before later effect expressions are evaluated (`crates/sembla-cuda/src/codegen.rs:1232-1268`). A double write can therefore mask a later effect overflow that CPU reports first. CPU also builds wires in wire declaration order and fields in field order (`crates/sembla-runtime/src/executor.rs:796-832`, `835-870`), while CUDA evaluates fields in parallel and reduces errors by flattened destination-input field index (`crates/sembla-cuda/src/codegen.rs:1290-1385`, `1424`; host launch at `crates/sembla-cuda/src/backend.rs:735-768`). Wire order need not equal destination box/port order, so two failing fields can yield a different first diagnostic. These are further valid-model violations of exact CPU error semantics.

### Other verified areas

- **Correct — reductions/scatters on non-error paths.** Grouped and wire-output `f64` sums use an ascending-row fold and a publication-only second stage (`crates/sembla-cuda/src/codegen.rs:945-988`, `1334-1382`), matching the CPU left fold (`crates/sembla-runtime/src/eval.rs:1556-1565`; `crates/sembla-runtime/src/executor.rs:862-869`). Effect writes are staged and scattered without result-bearing atomics (`crates/sembla-cuda/src/codegen.rs:1232-1286`).
- **Correct — deterministic/safe model identity and dump path.** Generated source contains a SHA-256 model-name label rather than raw model text and is content-hashed (`crates/sembla-cuda/src/codegen.rs:801-820`); the hostile-name and golden/determinism tests passed. Source dumping is content-addressed through `SEMBLA_CUDA_DUMP_DIR`.
- **Correct by static ABI audit; runtime ABI remains unanswered.** Host launch order and scalar widths match generated kernel signatures for transition, aggregate, effect, output, and Philox kernels. The CUDA-feature crate and all targets compile. Actual NVRTC compilation and device launch ABI were not exercised.
- **Correct — residency and hashes by inspection.** State, wire inputs, aggregates, candidates, and effect buffers are allocated once in construction and swapped across ticks; state/input downloads occur only in `download_hash` (`crates/sembla-cuda/src/backend.rs:382-401`, `770-803`). The CUDA hash walks box/table/attribute and box/port/field declaration order with the V1/V2 domains (`crates/sembla-cuda/src/backend.rs:1047-1126`), matching the runtime contract (`crates/sembla-runtime/src/state.rs:484-510`). Real device hash equality remains criterion 3 unanswered.
