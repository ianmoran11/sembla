## Review

### Correct

- **Prior blocker 1 is resolved: hostile validated model names no longer enter CUDA source.** `crates/sembla-cuda/src/codegen.rs:801-810` emits only a SHA-256 hex digest of `model.name`; the remaining generated symbol names are derived from numeric rule IDs (`crates/sembla-cuda/src/codegen.rs:994-1003`). The always-running regression includes newline, `#error`, carriage return, a trailing backslash, and Unicode, and verifies an ASCII-only digest label (`crates/sembla-cuda/src/codegen.rs:1730-1741`). No IR-validation narrowing was introduced.

- **Prior blocker 2 is resolved.** The attempted CPU reassociation is gone: the tracked diff no longer changes `crates/sembla-runtime/src/eval.rs` or `crates/sembla-runtime/src/executor.rs`. The strengthened CPU regression freezes the ascending left fold result for `[1e16, 1, -1e16, 1]` as `1.0` (`crates/sembla-runtime/tests/eval.rs:678-709`). CUDA grouped sums now use one device worker and a row-ascending fold (`crates/sembla-cuda/src/codegen.rs:945-980`), while publication is a plain assignment with no `+ 0` identity merge (`crates/sembla-cuda/src/codegen.rs:984-988`). Input sums likewise fold rows ascending (`crates/sembla-cuda/src/codegen.rs:867-916`), and wire-output sums fold ascending before storing bits and later copying them (`crates/sembla-cuda/src/codegen.rs:1334-1382`, `1386-1418`). This preserves both floating cancellation order and checked-integer `[MAX, 1, -1]` failure order.

- **The cited part of prior blocker 3 is resolved.** The host now processes each transition as aggregate-first-use, transition kernel, candidate error scan, then claim validation before advancing to the next transition (`crates/sembla-cuda/src/backend.rs:419-533`). Generated candidate errors are separated into complete guard-row and hazard-row scans (`crates/sembla-cuda/src/codegen.rs:1003-1022`), and claim resource/key columns are validated claim-major and row-ascending (`crates/sembla-cuda/src/codegen.rs:1025-1052`). Therefore transition 0's claim error wins over transition 1's guard error, matching CPU order at `crates/sembla-runtime/src/executor.rs:649-680`. Aggregate first use across transitions is also staged on the first using rule (`crates/sembla-cuda/src/codegen.rs:832-840`; `crates/sembla-cuda/src/backend.rs:419-465`), so an earlier scalar transition error is committed before a later transition's aggregate is built.

- **Generated/host ABI inspection found no argument-order mismatch in the changed kernels.** Buffer widths and counts agree for aggregate partials/errors, doubled candidate-error slots, and output partials/errors (`crates/sembla-cuda/src/backend.rs:210-260`). The changed host launch lists match the generated signatures for aggregate, transition/claim, and output stages (`crates/sembla-cuda/src/backend.rs:426-464`, `485-533`, `684-768`; `crates/sembla-cuda/src/codegen.rs:945`, `984`, `1003`, `1022`, `1026`, `1310-1424`). The checked-in generated SIR fixture also passed a local Clang C++14 stub syntax check. This is not a substitute for NVRTC/device execution.

### High

- **Residual same-transition and later-phase aggregate error ordering still does not match the CPU oracle.** CUDA builds every aggregate first used by a rule before evaluating any scalar part of that transition (`crates/sembla-cuda/src/backend.rs:419-465`), whereas CPU evaluates an expression tree left-to-right (`crates/sembla-runtime/src/eval.rs:856-858`) while evaluating guard, hazard, then claims (`crates/sembla-runtime/src/executor.rs:657-680`). A valid single guard whose left operand overflows scalar integer arithmetic and whose right operand first uses an overflowing aggregate will therefore report the aggregate on CUDA but the scalar error on CPU. The same phase-order risk remains for effect-only aggregates: CUDA builds all active effect aggregates before any effect expression (`crates/sembla-cuda/src/backend.rs:579-635`), while CPU evaluates winning transitions and effects in declaration order (`crates/sembla-runtime/src/executor.rs:720-757`). Output-only aggregates are likewise all rebuilt before output fields run (`crates/sembla-cuda/src/backend.rs:677-723`), while CPU discovers them field-by-field after each field filter (`crates/sembla-runtime/src/executor.rs:845-888`). These are observable first-error mismatches for validated models, so the broad “mirror evaluator edge cases exactly” requirement is not yet fully met.

- **Cross-box first-error order remains vulnerable even without aggregates.** CPU completes `stage_box`, including winning effects, for an earlier box before staging the next box (`crates/sembla-runtime/src/executor.rs:305-309`, `720-773`). CUDA stages guards/hazards/claims for all globally ordered transitions before running any effects (`crates/sembla-cuda/src/backend.rs:419-578`, `639-658`). Thus an earlier box's effect overflow can lose to a later box's guard overflow. This matters because multi-box execution is in scope and is an acceptance fixture.

### Medium

- **The new reduction differential omits the input-aggregate cancellation case.** `sequential_group_and_output_reductions_match_cpu` covers grouped and wire-output sums plus checked grouped integer overflow (`crates/sembla-cuda/tests/gpu_semantics.rs:509-547`), but there is no `[1e16, 1, -1e16, 1]` input-aggregate sum differential. The input helper appears correct by inspection, but the requested grouped/input/output regression set is incomplete.

- **The first-error regressions are device-only and remain unexecuted.** The cited claim-before-later-guard and earlier-scalar-before-later-aggregate tests are correctly present (`crates/sembla-cuda/tests/gpu_semantics.rs:549-589`), but both are `#[ignore]`. The always-running fixture test checks CPU behavior and generated metadata, not CUDA execution (`crates/sembla-cuda/tests/gpu_semantics.rs:310-471`).

### Recommendation

**REVISE.** The three specifically cited failures are fixed, including the claim-before-later-guard case and aggregate first use across transitions. However, demonstrable same-transition, effect/output aggregate-phase, and cross-box effect ordering gaps still violate exact CPU first-error semantics. Add ordered first-use/error surfacing for those phases, add a device input-sum cancellation regression, then run the documented real-GPU suite before changing criteria 2–4 from unanswered.
